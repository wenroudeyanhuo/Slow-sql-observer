# AI Execution Plan for OpenSpec V2

## Document Purpose

This document is written for AI-assisted planning and implementation.
It defines the project scope, boundaries, configuration model, sequencing, deliverables, and acceptance criteria for V2 of Slow SQL Observer.

This document should be treated as the source-of-truth requirements brief for the next version after the current V1 pipeline.

## How AI Should Use This Document

When an AI agent or OpenSpec workflow consumes this file, it should:

1. treat this document as the authoritative V2 requirements source
2. preserve all stated scope boundaries and non-goals
3. derive change proposals, design documents, specs, and tasks from this document instead of re-expanding the scope
4. prefer incremental evolution from the existing V1 implementation rather than redesigning the system from scratch
5. keep V2 single-source and slow-log-file-based unless a later change explicitly supersedes this document

If any future proposal conflicts with this document, the conflict should be surfaced explicitly instead of being silently absorbed into scope.

## Version Definition

V2 upgrades Slow SQL Observer from a runnable local demo into a source-aware slow SQL observation system that can be connected to a real MySQL slow query log in a production-like environment.

V2 must preserve the V1 end-to-end pipeline while making the system operationally clearer and more suitable for long-running use.

## Project Definition

Build a source-aware, single-source MySQL slow query log analysis system.

The system should:

- read slow query events from one real MySQL slow query log file
- distinguish clearly between the observed source and the analysis storage
- parse slow log events into structured records
- normalize SQL into reusable templates
- aggregate metrics by fingerprint
- persist raw and aggregated data in an analysis database
- expose source metadata and collector runtime status
- support basic raw-record retention for long-running usage
- provide APIs and a web UI for operating and exploring the analysis results

## Product Goal

Deliver a V2 that proves the product can be connected to a real MySQL slow log source and operated with a clearer production-oriented model:

1. source configuration and analysis storage configuration are explicitly separated
2. one real slow log source can be configured and ingested continuously
3. source metadata and collector runtime status are queryable
4. parsed records, fingerprints, and aggregate statistics remain available through APIs and UI
5. raw records can be retained for a configurable window instead of growing without bound
6. documentation is sufficient for a user to connect the tool to a real MySQL slow log

## Non-Goals for V2

Do not implement the following in V2:

- multi-source or multi-instance orchestration
- `performance_schema` ingestion
- direct protocol-level SQL interception
- binlog-based SQL collection
- cross-host agent coordination
- alerting and notification workflows
- endpoint tracing or trace-to-SQL correlation
- SQL optimization recommendation engine
- ClickHouse migration
- OpenTelemetry integration
- tenant or RBAC systems

## Scope Boundaries

### In Scope

- one observed MySQL source
- one slow query log file source
- one collector runtime
- one analysis database
- source metadata exposure
- collector status exposure
- raw-record retention controls
- one backend API service
- one frontend web application

### Out of Scope

- multi-source scheduling
- managed fleet deployment
- remote log shipping agents
- full APM capabilities
- distributed observability systems

## Core Model

V2 must model three distinct concepts:

1. observed source MySQL instance
2. slow query log file generated by that source
3. analysis database used by Slow SQL Observer

The conceptual flow is:

```text
source MySQL instance
    -> source slow query log file
    -> collector
    -> parser
    -> fingerprint
    -> aggregator
    -> analysis database
    -> API
    -> web UI
```

## Configuration Model

V2 must clearly separate source configuration from analysis configuration.

### Source Configuration

Required:

- `SSO_SOURCE_INSTANCE_NAME`
- `SSO_SOURCE_SLOW_LOG_PATH`

Optional:

- `SSO_SOURCE_DB_DSN`
- `SSO_SOURCE_TIMEZONE`
- `SSO_SOURCE_DESCRIPTION`

Rules:

- the source slow log file remains the primary ingestion channel
- `SSO_SOURCE_DB_DSN` is an optional metadata and validation connection, not a required ingestion channel
- the system must still work without `SSO_SOURCE_DB_DSN` if the slow log file is accessible

### Analysis Configuration

Required:

- `SSO_ANALYSIS_DB_DSN`
- `SSO_ANALYSIS_DB_SCHEMA`

Rules:

- this database stores parsed records, fingerprints, aggregate stats, checkpoints, source metadata, and collector runtime state

### Runtime Configuration

Fields should include at least:

- `SSO_SERVER_ADDR`
- `SSO_WEB_DIR`
- `SSO_COLLECTOR_POLL_INTERVAL`
- `SSO_RAW_RECORD_RETENTION_DAYS`
- `SSO_LOG_LEVEL`

### Backward Compatibility

V2 should support V1 configuration names for one compatibility cycle, but:

- new names must be preferred in documentation
- deprecated names should trigger warnings
- a migration path must be documented

## Deployment Assumptions

V2 must explicitly assume:

- the collector runs in an environment that can access the slow log file
- the best-supported deployment is on the same host as the observed MySQL instance, or on a host with mounted access to the slow log file
- the tool does not automatically fetch logs across hosts

V2 documentation must explicitly state the source-side prerequisites:

- slow query logging is enabled
- `log_output=FILE`
- the configured slow log path is correct
- the collector process has permission to read the file

## Required Modules

### 1. Source Configuration and Metadata

Responsibilities:

- represent the observed source as a first-class concept
- persist source metadata and configuration-derived identity
- enrich source metadata when optional source DB connectivity is available

Expected outputs:

- source metadata records
- current source identity for APIs and UI

### 2. Collector

Responsibilities:

- read slow log files incrementally
- track file identity, checkpoint state, and last processed offset
- emit complete raw event blocks
- retain incomplete trailing content until it becomes complete

Expected outputs:

- raw slow log event blocks
- updated collector runtime state
- updated checkpoints

### 3. Parser

Responsibilities:

- parse MySQL slow log blocks into structured records
- extract timestamp, database, user, host, query time, lock time, rows sent, rows examined, and SQL text
- tolerate missing non-core metadata fields where reasonable

Expected outputs:

- `SlowQueryRecord` domain objects

### 4. Fingerprint

Responsibilities:

- normalize SQL parameters
- collapse structurally equivalent SQL into stable templates
- compute repeatable fingerprint hashes
- identify SQL type and main table when possible

Expected outputs:

- `Fingerprint` objects
- normalized SQL text

### 5. Aggregator

Responsibilities:

- update aggregate metrics by fingerprint
- maintain count, avg, max, and total timing metrics
- maintain rows-sent and rows-examined metrics
- maintain first-seen and last-seen timestamps

Expected outputs:

- `FingerprintStats`

### 6. Storage

Responsibilities:

- persist raw records
- persist fingerprints
- persist aggregated stats
- persist collector checkpoints
- persist source metadata
- persist collector runtime status
- support raw-record retention cleanup

Expected outputs:

- successful writes
- queryable read models
- retention cleanup behavior

### 7. Runtime Status

Responsibilities:

- track last successful ingest time
- track last checkpoint position
- track latest collector error time and message
- expose collector health and source accessibility state

Expected outputs:

- collector status records
- source status read model

### 8. API

Responsibilities:

- serve overview statistics
- serve fingerprint list and detail
- serve raw sample records
- serve source metadata
- serve collector runtime status
- support basic filter, sort, and pagination behavior

Expected outputs:

- stable HTTP APIs

### 9. Web UI

Responsibilities:

- display source identity and collector status
- display summary metrics
- display ranked fingerprint list
- display fingerprint detail and sample SQL
- distinguish clearly between empty data and collector/source errors

Expected outputs:

- dashboard page
- fingerprint list page
- fingerprint detail page
- visible source/status context

## Suggested Domain Models

### Source

Fields should include at least:

- source id
- source instance name
- source slow log path
- source description
- source db dsn presence flag
- source host or version when available
- created at
- updated at

### CollectorStatus

Fields should include at least:

- source id
- collector state
- last successful ingest at
- last checkpoint offset
- last file identity
- last error at
- last error message
- updated at

### SlowQueryRecord

Fields should include at least:

- record id
- source id
- occurred at
- db name
- user name
- client host or ip
- raw sql
- normalized sql
- fingerprint hash
- query time
- lock time
- rows sent
- rows examined

### Fingerprint

Fields should include at least:

- fingerprint id
- fingerprint hash
- normalized sql
- sql type
- main table name
- first seen at
- last seen at

### FingerprintStats

Fields should include at least:

- fingerprint id
- total count
- total query time
- avg query time
- max query time
- total rows examined
- avg rows examined
- max rows examined
- total rows sent
- avg rows sent
- max rows sent
- last seen at

### CollectorCheckpoint

Fields should include at least:

- source id
- log file path
- file identity
- last offset
- updated at

## Retention Requirement

V2 must formally support raw-record retention.

Minimum requirement:

- raw slow query records can be retained for a configurable number of days
- the retention rule applies to `slow_query_records`
- fingerprints and aggregate stats remain available by default
- the cleanup mechanism and schedule must be documented

## Implementation Order

Follow this order strictly unless there is a strong reason to change it:

1. redefine configuration model into source, analysis, and runtime groups
2. introduce source metadata and collector status models
3. update storage schema and compatibility logic
4. update collector to operate against source-aware configuration
5. keep parser and fingerprint behavior stable while integrating source identity
6. add retention cleanup capability
7. expose new source and collector status APIs
8. update frontend pages to display source context and status
9. revise documentation for real slow log onboarding and migration from V1

## Acceptance Criteria for V2

V2 is complete when all of the following are true:

1. source configuration and analysis configuration are distinct and documented
2. the tool can ingest from one real slow query log path, not only the bundled sample log
3. the tool works without `SSO_SOURCE_DB_DSN` when the slow log file is accessible
4. the tool can use `SSO_SOURCE_DB_DSN` when present to enrich source metadata or validation
5. source identity and collector runtime status are available through APIs
6. the web UI displays current source identity and collector status
7. raw-record retention is configurable and implemented
8. V1 configuration users have a documented migration path
9. the system can be started locally or in a production-like environment with documented steps

## Design Rules for AI Implementation

- preserve single-source scope in V2
- do not convert V2 into multi-instance management
- keep slow log file ingestion as the primary acquisition mechanism
- treat `SSO_SOURCE_DB_DSN` as optional metadata support, not the primary ingestion path
- do not introduce `performance_schema` collection in V2
- prioritize operational clarity over speculative extensibility
- keep interfaces explicit between source, collector, storage, and API layers

## Future Extensions

These may be proposed after V2, but should not be merged into the V2 scope:

- multi-source management
- `performance_schema` collector
- direct database query interception
- cross-host log shipping agents
- alerting and notification
- source-side configuration automation
- SQL data masking policies beyond basic retention
- OpenTelemetry and tracing integration
